## Addendum K: The Sustainability Covenant

### K.1 Architectural Philosophy
The Teleportation Protocol guarantees that software built with it will be sovereign, deterministic, and long‑lived. However, without financial sustainability, even the most elegant architecture can stall. **Addendum K** introduces a formal mechanism to embed business and funding goals directly into the `.tela` source of truth, making sustainability as measurable and enforceable as the vector delta.

### K.2 The `sustainability` Block

A project may include an optional `sustainability` block at the top level of any `.tela` file (or in a dedicated `sustainability.tela`). This block defines the mission, sponsors, roadmap milestones, and licensing strategy.

**Grammar:**

```tela
sustainability "domain_name" "version" {
  mission: "string"
  sponsors: [ { name: "string", type: "grant" | "commercial_license" | "individual", amount: number, period: "YYYY-QQ", status: "pending" | "active" | "expired" }, ... ]
  roadmap: [ { milestone: "string", date: "YYYY-MM-DD", funding_required: number, revenue_target: number }, ... ]
  license: "string"
}
```

All fields are mandatory unless marked optional.

- **`mission`** – A concise description of the project’s purpose.
- **`sponsors`** – An array of sponsors, each with:
  - `name`: The sponsor’s name.
  - `type`: `grant`, `commercial_license`, or `individual`.
  - `amount`: The total financial contribution (optional if non‑monetary).
  - `period`: The time period covered (e.g., `"2026-Q2"`).
  - `status`: `pending`, `active`, or `expired`.
- **`roadmap`** – An array of milestones, each with:
  - `milestone`: Description of the milestone.
  - `date`: Target completion date (ISO 8601 date).
  - `funding_required`: Funds needed to achieve the milestone (optional).
  - `revenue_target`: Expected revenue from the milestone (optional).
- **`license`** – The licensing model (e.g., `"MIT OR Commercial"`, `"GPL-3.0"`, etc.).

### K.3 The `telac sustain` Command

A compliant implementation of `telac` **SHOULD** provide a `sustain` subcommand. It reads all `.tela` files in a given directory (or a single file), extracts the `sustainability` blocks, and outputs a human‑readable report.

**Usage:** `telac sustain [path]`

**Report Content:**
- Project mission.
- List of active sponsors with amounts and periods.
- Upcoming milestones (those with a date in the future).
- **Funding gaps:** Milestones whose date has passed but `funding_required` > 0 (or `revenue_target` not met). These are flagged with a warning.

The report is intended to be consumed by humans or by CI systems.

### K.4 Sustainability CI Workflow

To automate monitoring, a project **SHOULD** include a GitHub Actions workflow (or equivalent) that runs `telac sustain` regularly (e.g., weekly) and on every push to `main`. If a funding gap is detected, the workflow **SHOULD** open or update an issue with the report.

**Example Workflow (`.github/workflows/sustainability.yml`):**

```yaml
name: Sustainability Check

on:
  schedule:
    - cron: '0 0 * * 1'   # every Monday
  push:
    branches: [ main ]

jobs:
  sustain:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install Rust
        uses: dtolnay/rust-toolchain@stable
      - name: Install telac (from source)
        run: cargo install --path ./tela
      - name: Run sustainability report
        id: report
        run: telac sustain . > sustain_report.txt
      - name: Create issue if funding gap
        if: contains(github.event_name, 'schedule')   # only on scheduled runs
        uses: actions/github-script@v6
        with:
          script: |
            const fs = require('fs');
            const report = fs.readFileSync('sustain_report.txt', 'utf8');
            if (report.includes('funding gap')) {
              await github.rest.issues.create({
                owner: context.repo.owner,
                repo: context.repo.repo,
                title: '⚠️ Funding milestone approaching',
                body: report
              });
            }
```

### K.5 Integration with the Teleportation Workflow

- **New projects** **SHOULD** include a `sustainability` block as part of their initial `.tela` blueprint.
